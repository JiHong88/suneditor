# changes.md 작성 가이드

> `changes.md`는 현재 릴리즈 버전의 변경사항만 기록하는 파일입니다.
> **suneditor-demo** 데모 업데이트, **릴리즈 노트** 작성 시 참고 자료로 활용됩니다.
> 릴리즈 완료 후 내용을 비우고 다음 버전을 위해 새로 작성합니다.

---

## 목적

| 활용처             | 설명                                                      |
| ------------------ | --------------------------------------------------------- |
| **suneditor-demo** | 데모 페이지에 새 기능/변경사항 반영 시 변경 목록으로 활용 |
| **릴리즈 메시지**  | GitHub Release, npm publish 시 changelog 초안으로 활용    |

---

## 기록 시점

- **새 기능 추가** (`feat`) — 데모에서 보여줄 수 있는 기능
- **버그 수정** (`fix`) — 사용자에게 영향이 있는 수정
- **동작 변경** (`refactor`, `perf`) — 외부 동작이 달라지는 리팩토링/최적화
- **Breaking Change** — API, 옵션, 플러그인 인터페이스 변경
- **기록하지 않는 것**: 내부 코드 정리(`style`), 문서(`docs`), 테스트(`test`), CI/빌드 설정(`ci`, `chore`, `build`) 등 사용자에게 직접 영향이 없는 변경

---

## 작성 형식

```markdown
### feat

- 기능 설명 (`관련 파일 또는 플러그인`)
- 기능 설명 (#이슈번호)

### fix

- 수정 내용 (`관련 파일 또는 플러그인`)

### change

- 동작 변경 설명

### breaking

- ⚠️ 변경 내용 — 마이그레이션 방법
```

### 규칙

1. **현재 버전의 변경사항만** 기록 — 이전 버전 히스토리는 남기지 않음
2. **커밋 타입별 그룹**: `feat`, `fix`, `change`, `breaking`
3. **한 줄 요약** — 짧고 명확하게 (커밋 메시지 스타일)
4. 관련 **파일/플러그인명** 또는 **이슈 번호**를 괄호로 표기
5. **Breaking Change**는 `breaking` 섹션에 별도 기재하고 마이그레이션 방법 포함
6. 영어 또는 한국어 — 릴리즈 노트 언어에 맞춰 작성

---

## 릴리즈 시 처리

1. `changes.md` 내용을 GitHub Release 본문 / npm 릴리즈 노트에 복사
2. suneditor-demo 업데이트 목록으로 활용
3. **릴리즈 완료 후 `changes.md` 내용을 모두 비움** (파일은 유지, 내용만 삭제)
4. 다음 버전 개발 시 새로 기록 시작

---

## Agent 지침

Claude Agent가 코드 수정 후 `changes.md`를 업데이트할 때:

1. 현재 `changes.md`를 읽고 `[Unreleased]` 섹션 확인
2. 해당 커밋 타입 그룹(`feat`/`fix`/`change`/`breaking`)에 항목 추가
3. 그룹이 없으면 새로 생성
4. 중복 항목 확인 — 이미 기록된 내용은 추가하지 않음
5. `style`, `docs`, `test`, `chore`, `ci`, `build` 타입은 기록하지 않음
